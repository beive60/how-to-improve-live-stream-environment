# 1. Content Creation Strategy

## 1.1. コンテンツ制作ガイドライン（アルゴリズム対応）

### 1.1.1. メタデータ設計

#### 1.1.1.1. タイトル（60文字以内推奨）

- 主要検索キーワードを前半に配置する
- チャンネル名やシリーズ名はタイトルに含めない（検索クエリに一致しない文字列はCTRを下げる）
- 感情的な誇張（「衝撃」「ヤバい」）は使用しない。技術チャンネルの信頼性を毀損する

#### 1.1.1.2. 説明文（5,000文字まで）

- 最初の2行に動画の要約と主要キーワードを入れる（検索結果に表示される部分）
- タイムスタンプ（チャプター）を必ず記載する
- 関連動画へのリンクを含める
- SNSリンク、Twitchリンクを固定テンプレートとして末尾に配置する

#### 1.1.1.3. タグ（非推奨）

- YouTubeは2026年現在、タグをランキング要素としてほぼ使用していない
- ハッシュタグ（#）を3つまで使用する方が効果的

### 1.1.2. サムネイル設計テンプレート

全動画で一貫したビジュアルアイデンティティを維持するため、以下のテンプレートを適用する。

#### 1.1.2.1. レイアウト構造

- 左1/3: VTuberアバターの表情（驚き、困惑、解決後の笑顔等）
- 中央-右: 技術的なビジュアル要素（波形、グラフ、設定画面のクローズアップ）
- 上部または下部: テキスト（3-5語、太字、ドロップシャドウ付き）
- 背景: ドメインごとに固定色（オーディオ=青系、ビジュアル=紫系、ネットワーク=緑系、構成管理=橙系、DevOps=赤系）

#### 1.1.2.2. ドメイン別カラーコード

| ドメイン | 背景色（メイン） | アクセント色 |
| --- | --- | --- |
| AE: オーディオ | #1a237e（ダークブルー） | #42a5f5 |
| VT: ビジュアル | #4a148c（ダークパープル） | #ab47bc |
| NE: ネットワーク | #1b5e20（ダークグリーン） | #66bb6a |
| CM: 構成管理 | #e65100（ダークオレンジ） | #ffa726 |
| DO: DevOps | #b71c1c（ダークレッド） | #ef5350 |

### 1.1.3. 動画の長さに関する指針

| 動画タイプ | 推奨尺 | 理由 |
| --- | --- | --- |
| トラブルシューティング（Phase 1） | 10-15分 | 問題解決型は短めが好まれる。離脱を防ぎつつ8分以上は確保する |
| 体系的解説（Phase 2） | 15-25分 | 深い解説には時間が必要。ただし20分超はチャプター必須 |
| 構築/自動化（Phase 3） | 20-30分 | 実装手順を含むため長くなる。冗長な操作はカット/倍速で圧縮する |

## 1.2. IP資産活用戦略

オリジナルVTuberモデル（Live2D）および自己音声は、チャンネルの独自資産（Proprietary Asset）である。
これらを「装飾」ではなく「機能的ブランド要素」として最大限に活用する。

### 1.2.1. 音声制作パイプライン

#### 1.2.1.1. 基本方針: 自分の声で収録する

「人（ニン）の価値」を構築する以上、音声は自分の声であることが必須条件となる。
TTS（ずんだもん等）の使用は、以下の理由から採用しない。

- **ブランドの矛盾**: 「推測ではなく計測で語るエンジニア」が合成音声で喋ると、「人の価値」が毀損される
- **コモディティ化**: 無料TTSは解説動画市場で飽和状態にあり、差別化要素にならない
- **信頼性の低下**: 技術系コンテンツにおいて肉声は信頼性のシグナルとして機能する

#### 1.2.1.2. RVC（Retrieval-based Voice Conversion）の活用

自分の声を直接収録した上で、RVCを「品質補正・一貫性担保」のツールとして使用する。

#### 1.2.1.3. RVCの役割（TTS代替ではなく品質工具として）

- 収録環境の微細なノイズや体調による声質のブレを補正する
- チャンネル全体で一貫した音響特性（Sonic Signature）を維持する
- 声質の微調整により、聞き取りやすさを最適化する

#### 1.2.1.4. 自己音声モデルの構築手順

| Phase | 作業 | 要件 |
| --- | --- | --- |
| データセット作成 | ノイズのない環境で15-30分の朗読音声を収録 | コンデンサーマイク + オーディオIF（既存機材で可） |
| 前処理 | ブレス/リップノイズの除去、2-10秒単位に分割 | Audacity等の波形編集ソフト |
| 学習 | RVC v2でモデルをトレーニング | NVIDIA GPU（VRAM 8GB以上）、100-200エポック |
| 運用 | 収録した音声をモデルに通し、品質を均一化 | 推論はCPUでも可能 |

#### 1.2.1.5. コスト評価

| 項目 | コスト |
| --- | --- |
| ソフトウェア | 0円（オープンソース） |
| 計算リソース | 0円（既存GPU利用）- 数千円（クラウドGPU） |
| データセット作成 | 数時間の労働コスト |
| 機材 | 0円（既存のオーディオIF/マイクを流用） |

RVCモデルは一度構築すれば長期間使用可能な資産となる。
初期の労働コストに対し、音質の一貫性という継続的リターンが得られる。

### 1.2.2. ビジュアル資産の運用設計

#### 1.2.2.1. Live2Dモデルの運用方針

Live2Dモデルは動画内で「プレゼンター」として機能させる。ただし、フレーム単位の口パク同期は過剰品質であり、制作効率を著しく損なうため採用しない。

#### 1.2.2.2. ステートマシン的運用

モデルの表情を「状態（State）」として扱い、コンテンツのセクションに応じて切り替える。

| State | 表情 | 使用場面 |
| --- | --- | --- |
| Default | 通常/微笑 | 導入、一般的な説明 |
| Alert | 真剣/注意 | 重要なポイント、警告、バグ解説 |
| Thinking | 疑問/思考 | 問題提起、視聴者が躓きそうな箇所 |
| Success | 笑顔/自信 | 解決策の提示、まとめ |

これらの表情差分をLive2Dのパラメータで切り替え、高解像度PNGとして書き出して動画編集素材として使用する。セクションの変わり目やスライド切替のタイミングで差し替えるだけで運用可能。

#### 1.2.2.3. SDイラストの運用方針

Live2Dモデルは表情変化のみが可能で、表現に限界がある。
この制約を補完するため、SDイラストを使用する。

- 睨み
- どや顔
- にやけ顔
- リアクション
- 哀
- 引き
- 楽
- 喜

- 激怒
- 困惑
- 焦燥
- 照れ顔

- 大泣き
- 大笑い
- 怒
- 呆れ
- 余裕

### 1.2.3. サムネイルにおけるIP活用

サムネイルはCTRに直結する最重要ビジュアル要素であり、IPの活用効果が最も高い場所である。

#### 1.2.3.1. 活用方針

- Live2Dモデルのスクリーンショットを全サムネイルに配置する（視覚的アンカー）
- 表情はサムネイル用に誇張する
- モデルの視線をサムネイル内のテキストまたは技術要素に向けることで、視線誘導を行う
- モデルの主要色（髪色、目の色）をテロップや強調ハイライトのアクセントカラーとして使用し、ドメイン別カラーコード（セクション1.2）と組み合わせる

### 1.2.4. カラーパレット統一

モデルの主要色を動画内の全ビジュアル要素に適用し、「どこを切り取ってもこのチャンネルの動画である」という視覚的署名（Visual Signature）を確立する。

#### 1.2.4.1. マスターカラー・データシート

チャンネル全体の共通色（Visual Signature Color）として以下を定める。全セクション（セクション1.2、セクション2.3等）で参照される「モデルの主要色」はこの定義に準拠する。

#### 1.2.4.2. 基本数値（The Core Definition）

| 表色系 | 値 |
| --- | --- |
| HEX | #b35a7d |
| RGB | (179, 90, 125) |
| HSV | 336°, 50%, 70% |
| CIELab | $L^* = 50.11, a^* = 41.34, b^* = -3.22$ |

#### 1.2.4.3. アクセシビリティ・マトリクス

| 組み合わせ | コントラスト比 | 適合基準 |
| --- | --- | --- |
| 対 White (#FFFFFF) | $4.49:1$ | 18pt以上の太字、またはUI要素でAA準拠 |
| 対 Black (#000000) | $4.68:1$ | 標準テキストでAA準拠 |

Grayscale Check: $L^*$ 値が $50$ 付近の中明度であるため、背景が極端な白・黒のどちらであっても、境界線（Border）やシャドウによる補強が有効。

#### 1.2.4.4. 運用上の技術的推奨事項

- **ダークモード対応**: 背景が #121212 等の深暗色の場合、視認性が低下するリスクがある。その際は彩度を維持したまま明度を $5\text{--}10\%$ 上げたバリアント（例: #c46b8e）への動的切り替えを検討する
- **印刷（CMYK）**: マゼンタ（M）主体の発色となる。プロセスカラーでは C25 M75 Y30 K0 付近が近似値となるが、入稿時は必ず $L^*a^*b^*$ 値を基準とした色校正を行う

#### 1.2.4.5. 適用ルール

- テロップの文字色/背景色にマスターカラー #b35a7d を組み込む
- コードブロックのハイライト色をマスターカラーのアクセント色と統一する
- ドメイン別カラーコード（セクション1.2）はサムネイル背景に使用し、マスターカラーはチャンネル全体の共通色として使用する
